# Production Recommendation

For a production deployment, the next step would be to harden the crawler around durability, politeness, and operational visibility. The current single-node SQLite design is appropriate for a local exercise and small deployments, but in production I would move crawl frontier management and indexing into more explicitly durable components, such as a dedicated relational database or log-backed queue. I would also add robots.txt support, per-host rate limiting, retry policies with exponential backoff, content hashing for duplicate detection, and structured logging and metrics for crawl throughput, failures, and index freshness.

If search must remain available during active indexing at larger scale, I would keep the same incremental indexing model but separate crawling, storage, and query-serving responsibilities more cleanly. A practical next step would still be a single-machine deployment with a stronger storage layer and better recovery handling; after that, the architecture could evolve into a distributed design with partitioned crawl queues and a dedicated search backend if scale required it.
